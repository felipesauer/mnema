## Findings (frictions)

Frictions are numbered in discovery order. Severity tag: 🐞 bug,
🤔 design gap, 💡 polish.

### F-1 — Migrations 006/007/008 not auto-applied on this DB 🤔

**Observed:** After upgrading the global `mnema` binary to 0.3.0, the
first `skill_record` call returned `no such table: skills`. The DB
was at version 5 (no_006, no_007, no_008). The cooperative drift guard
introduced in MNEMA-12 blocks *task mutations* under drift, but does
not block read calls on tables that don't exist yet, so the failure
came as a raw SQLite error rather than a hint.

**Sub-fix needed:** the MCP server boot should call `detectDrift` and
log a clear warning (or refuse to register the affected tools) when
the schema is behind. Alternatively, the skill/memory/observation
tools should check for pending migrations and respond with a
`SCHEMA_OUT_OF_DATE` error like the CLI mutations already do.

**Workaround for users today:** run `mnema migrate` after upgrading.

### F-2 — `memory_record` `no_op` still bumps `updated_at` 🐞

**Observed:** Wrote `forward-only-migrations` twice with byte-identical
content. Service correctly returned `action='no_op'` and skipped the
`.md` mirror write, but the SQLite row's `updated_at` advanced from
`18:05:19.633Z` to `18:05:33.289Z`. Mirror frontmatter stayed at the
original timestamp, so DB and mirror now disagree.

**Root cause:** `MemoryRepository.upsert` always runs the SQL UPDATE
regardless of the service-level no_op detection. The service only
gates the mirror write, not the DB write.

**Fix:** in `MemoryService.record`, skip the `repo.upsert` call when
`action === 'no_op'` and return the existing row directly. Same shape
applies to `SkillService.record` when content is byte-equal under
`mode='update'`.

**Impact:** moderate. Agents reasoning "memories updated since X" will
see false positives. Audit still records the no_op kind correctly.

### F-3 — `skill_record` `mode='new_version'` resets usage_count silently 🤔

**Observed:** v1 had `usage_count=1`. Bumping to v2 created a new row
with `usage_count=0` and `last_used_at=null`. The tool description
doesn't mention this; an agent could expect counters to carry forward.

**Resolution path:** either (a) carry over `usage_count` to new
versions, or (b) document the reset explicitly in the tool description
and add a `previous_version_usage` field to the return payload. The
current behaviour is arguably *correct* for "validate adoption of new
version" but it's not signalled.

**Suggested action:** document in the tool description first; revisit
behavior only if real-world use shows confusion.

### F-4 — `skill_use` returns full content despite docstring 🐞

**Observed:** Tool description says "Does not return the content; call
`skill_show` for that." The implementation returns the full Skill row,
including `content`.

**Root cause:** `SkillService.recordUse` returns the updated entity;
the registrar wraps it in `ok({ skill: result.value })` without
projecting away the content field.

**Fix:** project the response in `skill-tool.ts` to omit `content`
(and optionally other heavy fields). Two options:
- minimal: `ok({ skill: { slug, version, usage_count, last_used_at } })`
- structured: introduce a `SkillSummary` shape and return that.

**Impact:** small but real — wastes tokens, contradicts docs. Agents
that trusted the docstring may have unnecessarily called `skill_show`.

### F-5 — `recent_observations` in bootstrap omits `id` 🤔

**Observed:** `context_bootstrap` returns `recent_observations` with
`{ content, topics, at }` only. The Observation entity has an `id` and
optional `relatedTaskId`. An agent cannot reference these observations
in a follow-up call (e.g. "consolidate observation X into a memory")
without first listing them via `observations_list`.

**Resolution path:** include `id` and `related_task_key` (resolved
back from `related_task_id`) in the projected shape. Marginal payload
cost, big UX win.

### F-6 — `memory_index` / `decisions_index` legacy paths return null with new memory rows 🤔

**Observed:** `context_bootstrap` still returns `memory_index: null`
and `decisions_index: null` even though `memories` table has 3 rows.
The legacy fields read `.mnema/memory/INDEX.md` and
`.mnema/memory/decisions/INDEX.md` (generated by
`mnema memory consolidate`), which the agent didn't touch.

**Tension:** two sources of truth for memory now exist: the legacy
human-curated INDEX.md and the new SQLite `memories` table. Agents
reading bootstrap will be confused when one is empty but the other
has content.

**Resolution:** either run `memory consolidate` automatically when
agents write memories (so INDEX.md mirrors SQLite), or drop the legacy
fields in a future bump and let `memories_inventory` be the single
inventory. Documenting the precedence in `SKILLS-AND-MEMORY.md` is the
minimum.

### F-7 — Doctor mirror-drift check reports green (`✓`) even with missing files 💡

**Observed:** After `rm .mnema/skills/add-mcp-tool.md`, `mnema doctor`
shows:

```
✓ skills mirrored  2 rows, missing files: add-mcp-tool
```

The detail line lists the drift, but the visual badge is `✓` (green
checkmark) like every other passing check. The plan called for
warnings, but the rendering layer has no warning level — only
ok/not-ok.

**Fix options:**
- minimal: change `ok: true` to `ok: false` for drift, accept that
  doctor exits non-zero. Risk: blocks projects that have stale
  mirrors after editing skills manually.
- proper: extend `DoctorCheck` with `severity: 'ok' | 'warning' | 'error'`
  and render `⚠` for warnings. Keep doctor exit 0 if only warnings.

**Recommendation:** the proper fix — drift is real signal, the
checklist has 12+ items and a `⚠` would stand out.

### F-8 — No way to rebuild mirror from SQLite via MCP 🤔

**Observed:** I deleted a mirror file to test drift detection. To
restore it I needed to write back. `skill_record` with byte-identical
content correctly returns `no_op` and **does not regenerate** the
mirror (the service's check `if (action !== 'no_op') writeMirror`
skips it). I had to restore the file manually via `Write` outside the
MCP surface.

**Resolution path:** a small admin tool like
`mnema memory consolidate` but for the SQLite-backed tables — call it
`mnema mirror rebuild` or surface `--rebuild-mirrors` on the existing
`doctor` command. Alternatively, `skill_record`/`memory_record` could
detect mirror absence as part of "content changed" and rewrite.

The "detect mirror absence and rewrite even on no_op" approach is
elegant: it makes the mirror the redundant copy, with SQLite always
authoritative. Implement at the service layer (check `existsSync` on
the expected path; force `writeMirror` if missing, even when no_op).

## What worked well

- **Task linkage via human key** — `observation_record(related_task_key='MNEMA-15')`
  resolved to UUID transparently. Round-trips correctly via
  `observations_list`. Identical UX to the `assignee` handle resolution
  from 0.2.0 — consistent.
- **Skill versioning model** — agent intuitively picked `mode='update'`
  for refinements and `mode='new_version'` when realising something
  bigger had changed. The `action` hint in the response is the right
  signal. v1 stayed queryable.
- **Frontmatter generation by service** — agents pass semantic fields,
  service writes valid YAML. No agent ever had to think about YAML
  escaping. The `quoteYaml` helper handled the colon-containing titles
  correctly.
- **Audit trail is complete** — every record/use call ended up in
  `.mnema/audit/current.jsonl` with the right kind and run id. Audit
  was the cheapest way to reconstruct what dev5 actually did, end to
  end. No friction here.
- **Bootstrap discovery loop** — dev5 noticed in observation 5 that
  "the path observation → consolidate into memory → harden into skill"
  feels intentional in the design, just from using the tools. The
  ergonomic gradient (lower friction = lower commitment) self-explained.
- **Doctor mirror count is right** — reports 2 skills mirrored (latest
  version per slug), not 3 (raw row count). Caller wants "how many
  slugs", and that's what it gets.

## What dev5 found non-obvious

Verbatim from sub-agent report (notable insights worth capturing):

> **Hierarquia implícita de "fricção pra registrar":** observation
> (zero discipline, append-only) → memory (slug + título, upsert) →
> skill (slug + name + description + content + tools_used + version).
> O agente é naturalmente empurrado pra escolher o artefato pelo custo
> cognitivo, não pela importância. Isso bate exatamente com o pipeline
> "observation vira memory vira skill" — o design recompensa o caminho
> certo.

> **`UNIVERSAL_TOOL_NAMES`** is the "canonical index" that
> `mnema skill lint` reads. Skills referencing tools outside that list
> are marked invalid — the lint doesn't talk to the server, it reads
> the constant. This means lint can give false negatives if the
> registry is out of date relative to the booted server.

The first point is a confirmation that the design intent translated
into the UX. The second is a real edge case worth a follow-up
("MCP server smoke test in `mnema doctor` should compare boot tools
to registry constant").

## Recommendation

8 frictions found, none blocking. Of the 8:

- **F-1, F-2, F-4** are correctness issues — schedule for 0.3.0-alpha.1.
- **F-7, F-8** are polish — schedule for 0.3.0-alpha.1 if cheap, defer
  otherwise.
- **F-3, F-5, F-6** are design gaps — discuss and either document or
  fix. F-6 is the largest in scope (legacy memory_index coexisting
  with new memories table).

Suggested order for 0.3.0-alpha.1:
1. F-4 (1-line projection in skill-tool.ts)
2. F-2 (skip repo.upsert on no_op in MemoryService + SkillService)
3. F-1 (warn or refuse tool calls when DB is behind disk migrations)
4. F-5 (project `id` + `related_task_key` into bootstrap observations)
5. F-8 (rewrite mirror on no_op if file is missing — defensive)
6. F-7 (add `severity` to DoctorCheck — small refactor but a polish win)

F-3, F-6 ride along in `SKILLS-AND-MEMORY.md` as documented
clarifications rather than code changes for this bump.

The feature itself is solid: agents recorded knowledge across 3
shapes, mirrors materialised correctly, bootstrap inventories sang,
doctor saw drift. The pipeline observation → memory → skill emerged
without coaching. Ship 0.3.0-alpha.1 with F-1..F-5 fixed and we have
a sound public alpha to socialise this design.
