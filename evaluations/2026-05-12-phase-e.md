repacked after the Camada-3 sweep so the global binary carries
migrations 009 + 010)
**Tester:** felipesauer (PO) + sub-agent `dev7` (CLI-only, no MCP)

## Findings (8 total)

Severity tag: 🔴 high, 🟡 medium, 🟢 low.

### F-E1 🔴 → fixed
**FTS query with reserved characters leaked a `SqliteError` stack-trace.**
Any `mnema search` with `;`, unmatched quote, raw `AND OR`, or `*`
prefix on its own crashed the CLI with exit 1 and printed 20+ lines
of stack including absolute `node_modules` paths. Not SQL injection
(`MATCH ?` is a bound parameter), but a confidence and UX break.

**Fix:** `SearchService.search` now returns
`Result<SearchHit[], MnemaError>`. The `try/catch` maps `fts5:` /
`MATCH` syntax errors to `ErrorCode.SearchInvalidQuery` with the
offending query + detail; SQLITE_BUSY surfaces as
`ErrorCode.StorageBusy` (see F-E4). The CLI prints a one-line hint;
the MCP path returns the structured error. Test added in
`tests/integration/services/search-service.test.ts:F-E1`.

### F-E2 🔴 → fixed
**Orphan mirrors (FS→DB drift) were invisible.**
`.mnema/skills/ghost.md` with no matching SQLite row neither showed
up in `mnema doctor` nor got cleaned by `--rebuild-mirrors`. In
production this means renaming a slug or deleting a row via SQL
silently leaves an orphan file forever.

**Fix:** `inspectMirrorDrift` now scans both directions. The
`skills mirrored` / `memories mirrored` checks list both missing
files and orphan files in their detail. New flag
`mnema doctor --rebuild-mirrors --prune-orphans` deletes the orphan
files; without `--prune-orphans` the orphans are reported but
preserved (safe default — humans might be editing the file). Tests
added in `tests/integration/cli/doctor-drift.test.ts:F-E2`.

### F-E4 🟡 → fixed (same commit as F-E1)
**`SQLITE_BUSY` during concurrent CLI mutations leaked stack-trace.**
Same root cause as F-E1 — unhandled `SqliteError`. The new
`SearchService.search` `try/catch` maps it to
`ErrorCode.StorageBusy`. Wider mutation paths (task move, sprint
transition) still propagate raw errors when racing — those still
need wrapping at the service boundary. **Partial fix**; tracked.

### F-E3 🟡 → trade-off documented
**CLI does not expose `--expected-updated-at` for `decision
accept/reject/supersede` and `sprint start/close`.**
Camada-3 delivered service + repository support for the
optimistic-concurrency token but no CLI surface. Two humans accepting
the same ADR via CLI still race (last write wins).

**Decision for this cut:** acceptable trade-off. Optimistic
concurrency is most useful when programmatic clients (agents, the
MCP tool) coordinate. The CLI is dominated by single-user
interactive sessions where the read→decide→accept window is short
enough that the race is rare. If field reports show otherwise, add
the flag to `decision accept/reject/supersede` and
`sprint start/close`. Tracked in `docs/TECH_DEBT.md`.

### F-E5 🟡 → not changed
**Workflow `kanban` declares `features.sprints: false`, but
`SprintService.plan` still creates sprints.**
`features.sprints` is documented as an opt-in for workflows. The
service does not enforce it. dev7 created `PHE-SPRINT-1` in the
kanban project without any warning.

**Decision:** ambiguous design. The flag could mean "this workflow
doesn't enforce sprint concepts in its transitions" (current truth)
or "this workflow forbids sprints entirely" (what the name suggests).
Document the former in workflows JSON description, or enforce. For
now: deferred to TECH_DEBT.

### F-E6 🟢 → not changed
**`mnema task move <key> <action> field=value` accepts unknown fields
silently.** Typos like `assigneed_id=...` are no-ops.

**Decision:** the gate validates *required* fields. Strict-mode
field rejection would be a behaviour change (existing flows pass
extra annotation fields that are intentionally tolerated by the
service to keep MCP payloads forward-compatible). Tracked but no
fix in this commit.

### F-E7 🟢 → fixed
**`mnema decision show` missing `--json` flag.** Parity gap with
`task show`, `sprint show`. Fixed by adding the flag and the same
serialisation pattern used by sibling commands.

### F-E8 🟢 → tracked
**`mnema audit query` cannot filter by entity key** (only by `kind`,
`actor`, `via`, `run`). Workaround: pipe to `jq`. Tracked.

## What worked well (no friction)

- **Workflow `kanban` end-to-end**: promote/start/block/unblock/complete
  all ran via CLI. `tasks_list --state BLOCKED` accepted the value
  (the workflow-as-data fix from the previous commit). `mnema inbox`
  surfaced the BLOCKED queue correctly (the InboxService fix in the
  same sweep).
- **Optimistic concurrency at the service layer**: dev7 used `node`
  to invoke `DecisionRepository.updateStatus` with a stale token —
  got the expected `CONFLICT` result. Service path is solid even
  without CLI exposure.
- **FTS unicode61 with `remove_diacritics 2`**: `café` ↔ `cafe`
  matched; `日本語` matched exact. Volume 100+ observations responded
  in ~170 ms.
- **`--rebuild-mirrors` selectivity** (still): three runs in
  sequence reported "nothing to rebuild" on the second and third,
  exactly as Phase D expected.
- **Drift warning loud** still fires on MCP boot when migrations are
  pending (verified verbatim).

## Recommendation

Four findings fixed (F-E1, F-E2, F-E4 partial, F-E7), three tracked
as TECH_DEBT (F-E3 CLI surface for concurrency token; F-E5 sprints
feature flag enforcement; F-E6 strict field validation; F-E8 audit
filter expansion).

Phase E confirms the optimistic-concurrency work from Camada-3 is
solid at the service layer and exercised the third workflow shape
(kanban) end-to-end. The remaining mutation paths that need
`SqliteError` wrapping (task transition, sprint transition,
note add, etc.) are a small refactor worth picking up before
expanding the public surface — leaving raw `SQLITE_BUSY` to leak in
those paths is the same UX failure F-E1 fixed for search.
