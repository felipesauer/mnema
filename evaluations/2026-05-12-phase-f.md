## Findings — 12 total

### F-F1 🔴 → fixed
**Attachment metadata duplicates when the same content is re-attached.**
The `FileStore` content-addresses binaries (one blob per SHA-256), but
`AttachmentService.attachToTask` / `attachToDecision` always called
`repository.insert`, producing two rows pointing at the same blob, the
same parent, the same hash.

**Fix:** new `AttachmentRepository.findByParentAndHash`. The service
now returns the existing row when one matches, and the audit event
carries `deduplicated: true` so the agent's intent is still visible.

### F-F2 🔴 → fixed
**`attachments.path` baked in `.app/attachments/` prefix.** That path
does not exist on disk — files live at
`<paths.state>/attachments/<sha>.ext`. Constant left over from an
earlier layout. A project that moved its `.mnema/` directory would
break the lookup.

**Fix:** `FileStore.store` now returns `relativePath = filename`
(just `{sha256}.{ext}`). The doc-string on
`AttachmentRepository.Attachment` explains the convention; consumers
join with the configured state dir.

### F-F3 🔴 → fixed
**`workflow.terminal[]` overrode declared exit transitions.**
`jira-classic.json` declares `CLOSED.reopen → REOPENED` but the
engine treated CLOSED as a wall: `TERMINAL_STATE` was returned before
the workflow's own transition table was consulted. `default.json`
had the same issue with `DONE.reopen`.

**Fix:** `TaskService.transition` now checks
`stateMachine.listActionsFrom(state).length === 0` before refusing
with `TERMINAL_STATE`. A state is "terminal" only when no exit
transitions exist; declaring one wins.

### F-F4 🔴 → fixed
**`tasks.reopen_count` never incremented; never rendered.**
The column existed in the schema but no service path touched it. The
CLI formatter ignored it. Reopens were invisible after the audit log
rotated.

**Fix:** `TaskRepository.incrementReopenCount` bumps the counter;
`TaskService.transition` calls it whenever `input.action === 'reopen'`.
The CLI formatter prints `reopened: Nx` in the meta line when N > 0.

### F-F5 🟡 → not changed
**Markdown importer leaves `## DRAFT <title>` as the literal title.**
Confirmed in earlier phases as deliberate. The Phase B' eval doc
records the reasoning: a "state-blind" importer never silently moves
a task across workflow gates. The friction is real — agents need to
clean the prefix manually — but the safer behaviour wins. Doc it in
the import command help.

### F-F6 🟡 → tracked
**Markdown importer is not idempotent.** A second invocation
duplicates everything. No `--skip-existing`, no hash-of-heading
dedup. Comment in `markdown-importer.ts` documents the trade-off.
Acceptable for one-shot adoption flows; needs a flag if importer is
to become reusable.

### F-F7 🟡 → tracked
**`mnema attach` CLI only accepts task parents.** Schema and service
layer support decisions and notes too. Error message under decision
key is misleading: `Task PHF-ADR-1 not found`. Tracked as CLI surface
gap.

### F-F8 🟡 → tracked
**`mnema agent inspect` does not render parent/child runs.**
`agent_runs.parent_run_id` exists in the schema; the CLI ignores it.
Hierarchy reconstruction has to go through `audit query` instead.
Tracked.

### F-F9 🟡 → tracked
**`mnema destroy --help` advertises "two confirmations" but only
exposes `--yes`.** UX inconsistency with help text. Tracked.

### F-F10 🟡 → tracked
**`workspace_config.mode` accepts `multi` but no code consumes it.**
The schema documents the field but the multi-project flow has no
adoption path. Tracked — feature is inert until a multi-project
narrative emerges.

### F-F11 🟢 → tracked
**`destroy` leaves `.mnema/` and `.mnema/workflows/` empty dirs +
`.gitignore` intouched.** Minor cleanup.

### F-F12 🟢 → tracked
**GitHub importer hardcodes `https://api.github.com`.** No way to
point it at a mock for tests. Tracked for when the importer needs to
be unit-tested or used with GHES.

## What worked well (no friction)

- **`jira-classic` workflow path**: OPEN → IN_PROGRESS → RESOLVED →
  REOPENED → IN_PROGRESS → RESOLVED → CLOSED all worked. The new
  `CLOSED.reopen` path (post-fix) brings the task to REOPENED with
  `reopen_count: 1`.
- **Audit query filters** (`--kind`, `--actor`, `--since`, `--run`)
  all worked.
- **Agent run depth limit**: SQL bypass attempted depth 6 → CHECK
  constraint refused. The structured `DepthLimitExceeded` error
  shape exists; only the CLI doesn't expose nested-run creation.
- **Importer creates the right state on jira-classic**: tasks land
  in OPEN, audit emits `task_created`, mirror `.md` is generated.
- **`mnema destroy --yes`** removed the right tree and left only
  empty dirs.
