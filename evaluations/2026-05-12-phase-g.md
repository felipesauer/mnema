# Audit-log integrity — adversarial sweep + remediation

**Date:** 2026-05-12
**Mnema version under test:** 0.3.0-alpha.1 (commit `e94bfba`, repacked
after the medium-severity sweep so the global binary carries the
latest CLI and migrations 1–10).
**Tester:** felipesauer (PO) + sub-agent `dev9` (CLI-only, no MCP).
**Project under attack:** `/tmp/mnema-phase-g/` — synthetic project,
workflow `default`, 17 legitimate events seeded across 3 actors
(`daniel`, `eve`, `mallory`).
**Scope:** adversarial sweep against the JSONL audit log under
`.mnema/audit/`. The threat model: a process with write access to
the filesystem (the agent itself, or anyone with shell on the box)
tampering with the log to hide tracks.

---

## Result — 9 attacks, 9 silent successes

dev9 tried nine distinct tampering vectors against `current.jsonl`
and `YYYY-MM.jsonl`. **All nine succeeded without `mnema doctor`
flipping a single warning.** The full report is in
`/tmp/mnema-phase-g/PHG-FINDINGS.md`; summarised here:

| Vector | Severity | Defense before | Defense after |
|---|---|---|---|
| Edit a line (flip `actor`) | 🔴 | none | hash chain breaks |
| Forge an appended event | 🔴 | none | hash chain breaks |
| Delete `current.jsonl` | 🔴 | none | event count mismatch |
| Truncate the tail | 🔴 | none | event count mismatch |
| Reorder lines | 🔴 | none | hash chain breaks |
| Replay (duplicate a line) | 🔴 | none | event count or chain breaks |
| Insert malformed JSON | 🟡 | silently swallowed | flagged as warning |
| Rename `YYYY-MM.jsonl` to another month | 🟡 | none | tracked (deferred) |
| `audit query` auto-sort masks reorder | 🟢 | by design | unchanged (UX trade-off) |

The full classification reasoning is in `PHG-FINDINGS.md` lines 13-65.

## What we shipped

### Hash chain on every event (schema `v: 2`)

Each line carries `prev_hash` (`null` for the first event in the
file) and `hash` (SHA-256 of the line with `hash` omitted). The
writer fills both in before `appendFileSync`. A second writer call
reads the latest `chain_head_hash` from SQLite and uses it as the
next `prev_hash`, so the chain spans rotations across files. Volume
is tiny — audit lines are bounded — so the per-write cost is
negligible.

### `audit_state` mirror in SQLite (migration 011)

A single-row table with `event_count`, `last_event_at`, and
`chain_head_hash`. Updated atomically with every `AuditWriter.write()`.
Lets `doctor` compare against what is on disk: a smaller count means
truncation/deletion, a mismatched tail hash means edit/forge, a
mismatched `prev_hash` means reorder.

### `mnema doctor` walks the chain

New `inspectAuditIntegrity(adapter, auditDir)` returns up to four
rows:
- `audit event count` — parseable lines on disk vs `event_count`
- `audit hash chain` — recomputed tail hash vs stored
- `audit lines parse` — count of unparseable lines as a warning
  (the smokescreen vector)
- `audit integrity: legacy` — warning shown when the project predates
  this feature; the integrity check activates on the next write

### Strict reader (opt-in)

`AuditQuery.runStrict()` returns the same events as `run()` plus a
diagnostic block (`malformedLines`, per-file breakdown). Doctor uses
this; CLI/MCP read paths keep the existing tolerant `run()` to avoid
breaking flows that read partial logs during normal rotation.

### Backward compatibility

The writer accepts a `state: AuditStateRepository | null` argument.
When `null` (legacy callers, unit tests), behaviour is the pre-G v1
shape — no chain, no mirror update. When a real repository is wired,
the v2 chain activates. Doctor reports "legacy audit log" until the
first event under the new writer lands; from then on the integrity
check is live.

## What we deliberately did not ship

- **F-G8 archive-rename invariant** — checking that `YYYY-MM.jsonl`
  filename matches the month of every `at` inside it. Cheap to add
  later; not on the critical path. Tracked.
- **F-G9 reorder masking by auto-sort** — keeping the in-memory
  sort because the CLI rendering convention (chronological output)
  matters more than surfacing on-disk order. The hash chain already
  catches the underlying tampering; the sort just makes the surface
  look tidy.
- **Hash chain across files** is implemented via the `audit_state`
  mirror (`chain_head_hash` survives rotation). The chain inside a
  single file uses `prev_hash`; across files, the writer reads the
  global head from SQLite for the first line of the new file. This
  was the cheaper design than embedding cross-file linkage in
  every line.

## What this does NOT prove

- A user with **SQL** access to the DB can still tamper with both
  the log and the mirror in lockstep. The hash chain raises the
  bar — it requires understanding the chain layout, recomputing
  every `prev_hash` and `hash`, and rewriting `chain_head_hash`
  atomically — but it does not make audit forensically signed. A
  future hardening would copy the `chain_head_hash` to an external
  store (signed by a key the local process does not hold).
- The CLI assumes a single writer. Concurrent writers would race
  on `audit_state.read() → write → recordEvent()` and could leave
  the chain inconsistent. A future fix would wrap the trio in a
  SQLite transaction. Tracked.
