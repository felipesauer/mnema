## Findings (1 high + 4 SMOKE.md typos)

### F-S1 🔴 `mnema migration generate` writes to the globally installed package

**What I did.**

```bash
cd /tmp/mnema-smoke
mnema migration generate smoke_demo
```

**What Mnema reported.**

```
✓ ../../home/felipe/.nvm/versions/node/v24.14.0/lib/node_modules/@saurim/mnema/dist/storage/sqlite/migrations/012_smoke_demo.sql
  next time the database is opened, version 12 runs and stamps schema_migrations.
```

The stub was written to the **globally installed Mnema package**
(`~/.nvm/versions/.../@saurim/mnema/dist/storage/sqlite/migrations/`),
not under the user's project. Consequences:

1. Cross-project contamination — every Mnema project on this machine
   would inherit `012_smoke_demo.sql` on next boot.
2. Reinstalling Mnema (`npm i -g`) wipes the generated migration
   silently — the user thinks the migration is durable.
3. Migration count drifts vs the package's bundled set; the next
   real migration shipped upstream collides on version 12.

**Root cause.**
`src/cli/commands/migration-command.ts:39` calls
`generateMigration(migrationsDir(), slug)`. `migrationsDir()` lives
in `src/utils/asset-paths.ts` and resolves to the package root — the
correct path for **reading** bundled migrations, but the **wrong
path for writing** user migrations.

**Severity 🔴.** Data layout violation; user expectation completely
broken. Functional path for migrate (no-op when clean) still works,
but the generator is unusable as shipped.

**Suggested fix.** Resolve the project root via `resolveProjectRoot`
(same pattern doctor and the rest of the CLI use), default the
generator to `<project_root>/.mnema/migrations/`, and update
`MigrationRunner.run` to merge bundled + project-local migration
files at boot. Alternatively, accept `--out <dir>` and refuse to
write unless the user opts in to a path.

**Cleanup performed during the smoke.**
```bash
rm /home/felipe/.nvm/versions/node/v24.14.0/lib/node_modules/@saurim/mnema/dist/storage/sqlite/migrations/012_smoke_demo.sql
```
Confirmed the package directory is back to migrations 001–011 only.

### F-S2 🟢 `acceptance_criteria` requires a comma to be parsed as an array

**What I did.** `mnema task move SMK-2 submit ... acceptance_criteria="Bench beats current" estimate=3`.

**Observed:** `gate validation failed — acceptance_criteria: expected
array, received string`.

**Root cause.** `coerceValue` in `task-command.ts:248` only splits on
`,`. A single-criterion submit needs either `criterion="X,"` (which
the filter drops empty) or the multi-arg trick
`acceptance_criteria="A" acceptance_criteria="B"` (Commander dedups).

**Severity 🟢.** Documented contract — payload parsing on the CLI is
"value with comma → array; without → string". Confusing UX but not
a bug. SMOKE.md should always use 2+ criteria separated by comma to
sidestep this; the docstring of `task move --help` could call it
out more clearly.

### F-S3 🟢 `decision show` renders `Superseded by:` with internal UUID

**What I did.**

```bash
mnema decision supersede SMK-ADR-1 --by SMK-ADR-2
mnema decision show SMK-ADR-1
```

**Observed:** the `Superseded by:` line carries the decision's
internal UUID (`019eaa42-9d64-…`), not the human key `SMK-ADR-2`
the caller passed in.

**Severity 🟢.** Cosmetic; the UUID is correct, the human key would
just be friendlier. Easy fix in
`src/cli/formatters/decision-formatter.ts` (resolve via repo lookup
before rendering).

### F-S4 🟡 `mnema doctor --rebuild-mirrors` doesn't recreate the `memory/` tree

**What I did.**

```bash
rm -rf .mnema/skills .mnema/memory
mnema doctor --rebuild-mirrors
# ✓ nothing to rebuild — every row already has a mirror
mnema memory consolidate
# — memory: not initialised
```

**Observed.** `--rebuild-mirrors` only restores `.md` files for
SQLite-backed skills and memories; it doesn't recreate the empty
`.mnema/memory/decisions/` and `.mnema/memory/notes/` directories
that `memory consolidate` expects. The next `memory consolidate`
run prints `not initialised`.

**Severity 🟡.** `mnema adopt memory` or `mkdir -p` recovers
manually. Worth adding to `--rebuild-mirrors` so doctor's recovery
flow is complete.

### F-S5 🟡 Schema-out-of-date error fired briefly during a successful run

**What I did.** Two `mnema task move` invocations back-to-back during
Phase 6. First one returned:

```
error: Schema is out of date — 3 migrations pending
  - 009_fts_skills_memories_observations.sql
```

The second invocation (a few seconds later) succeeded. `mnema doctor`
reports `11 applied, 11 on disk` — the database **is** up to date.

**Severity 🟡 (could not reproduce on demand).** Possible causes:

- Race between two CLI invocations during the bash sequencing —
  the second process opened the DB before the writer fully
  committed migration 011's INSERT.
- Stale schema-version cache inside the SQLite connection (the
  migration runner reads `schema_migrations` lazily).

**Suggested next step.** Add a `--debug` flag to the CLI that dumps
the migration runner's view of applied vs available so the user can
attach the diff when this surfaces again. Or wrap the
`requireFreshSchema` check in `withMutatingCliContext` with one
retry on `STORAGE_BUSY`.

## SMOKE.md typos to fix (cosmetic, no functional impact)

While running, three small inconsistencies between the SMOKE.md
script and actual CLI behaviour surfaced:

1. **Phase 6 — `acceptance_criteria="Bench beats current"`** — single
   value without comma is rejected by gate. Make every example use
   2+ criteria separated by comma.
2. **Phase 7 — `mnema search "redis" --entity task,note`** — Commander
   `--entity <kind...>` is a variadic flag; the comma form is
   rejected with `unknown --entity value(s): task,note`. Replace
   with `mnema search "redis" --entity task note` (space-separated).
3. **Phase 18 — `ls -d skills memory roadmap`** — `mnema adopt all`
   creates these dirs under `.mnema/`, not at project root. Replace
   with `ls -d .mnema/skills .mnema/memory .mnema/roadmap` and the
   subsequent `skills/my-custom.md` write with
   `.mnema/skills/my-custom.md`.

These don't block the smoke (a careful reader self-corrects) but
slow down a first-time user.

## What worked well

- **Doctor's integrity surface is sharp.** Adding one fake event to
  `2026-04.jsonl` flipped `audit event count` to `disk has 30,
  audit_state has 29` immediately. After deleting the file, the row
  went back to `✓ 29 events match`. Hash chain stayed valid
  throughout (the manual file was a v1-shaped event, so the chain
  walk skipped it).
- **Workflow refines from F-GW1.** Mutating `transitions.DRAFT.submit.to`
  to `PHANTOM` produced exactly the expected `✗ workflow loads —
  transition 'DRAFT.submit' points to unknown state 'PHANTOM'` line.
- **Migration drift detection.** `mnema doctor` reports `11 applied,
  11 on disk` after every state-mutating phase. The chain head hash
  advances visibly between runs.
- **Hash chain in JSON output.** `mnema history --since 24h --json`
  shows `v: 2`, `prev_hash`, and `hash` on every event from
  migration 011 onwards. Useful for external integrity audits.
- **Bootstrap script** (`pnpm smoke:bootstrap`) worked first try —
  wipes `/tmp/mnema-smoke`, recreates, prints version. Took under
  a second.
