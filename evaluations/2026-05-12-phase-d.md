# Phase D — stress test of 0.3.0-alpha.1 post-release features

**Date:** 2026-05-12
**Mnema version under test:** 0.3.0-alpha.1 (commit `561d3c0`, repacked
after the Camada-2 sweep so the global binary carries migration 009)
**Tester:** felipesauer (PO) + sub-agent `dev6` (CLI-only, no MCP)
**Project under management:** `/tmp/mnema-phase-d/` — synthetic seed
project, workflow `default`. A side project at `/tmp/mnema-drift-test/`
exercised the MCP boot drift scenario.
**Scope:** features introduced *after* 0.3.0-alpha.0 — the Camada-2
sweep (drift warning loud, FTS5 for skills/memories/observations,
`mnema doctor --rebuild-mirrors`) plus cross-flow scenarios not
covered by Phase C.

---

## Setup

1. `pnpm pack` against the post-Camada-2 working tree built a `0.3.0-alpha.1`
   tarball. **Process finding** below: a previous `0.3.0-alpha.1`
   tarball already existed from before migration 009 landed. Same
   version, different content.
2. `npm i -g` installed the repacked global. `mnema migrate` in
   `/tmp/mnema-phase-d/` applied migration 009 cleanly.
3. dev6 spawned via `Task` tool with CLI-only access — by design,
   exercising the human path that does not run an MCP server.

## What dev6 produced

Five scenarios (A drift warning, B FTS5, C rebuild-mirrors, D obs
volume, E decision↔memory cross-search). Two confirmed clean, one
clean with caveats, one fully broken at the CLI surface, one mostly
clean with friction. Six new frictions surfaced.

## Findings

### F-D1 🔴 `mnema search --entity skill|memory|observation` silently returns nothing

**Observed:** `mnema search "bcrypt" --entity skill` returns `(no
matches for "bcrypt")` even when two skill rows contain the term and
`mnema search "bcrypt"` (no filter) returns them.

**Root cause:**
[src/cli/commands/search-command.ts:34-37](src/cli/commands/search-command.ts)
filtered the user-supplied entity list through a hardcoded
`['task', 'note', 'decision']` whitelist before passing it to the
service. Phase C added the three new entities to `SearchService`,
0.3.0-alpha.1 wired the MCP tool, **but the CLI shim was never
updated** — so `--entity skill` collapsed to `[]`, which the service
then treated as "no allowed entities → no hits".

**Severity:** high. Defeats the entire FTS feature for the new tables
when filtered via CLI. The MCP `tasks_search` tool was fine because it
ran its own zod enum (which the Phase C sweep updated). The CLI was
the orphan.

**Fix:** updated the CLI to use the canonical `SearchEntity` type
from `search-service.ts`. Unknown values now exit `2` with
`error: unknown --entity value(s): ...`. Help text now lists all six
kinds. Applied in this commit.

### F-D2 🟡 `mnema search` memory snippet returns the slug instead of the body

**Observed:** A memory with `slug=bcrypt-cost` and a substantial
`content` body matched on `"bcrypt"`. The CLI rendered:

```
memory bcrypt-cost bcrypt cost 12 chosen over argon2
  bcrypt-cost
```

The snippet field carried the slug — not useful for the human reader.

**Root cause:** `searchMemories` used `snippet(memories_fts, -1, ...)`
which lets FTS5 pick *any* indexed column. Slugs are dense and short,
so the ranker often prefers them when the term matches there.

**Severity:** medium. Renders the search hit useless without a
follow-up `memory show`.

**Fix:** force the snippet to come from `content` (column index 3),
with a fallback to `title` (column 2) if the match was only in
slug/title and content has no excerpt. Same treatment applied to
`searchSkills` (now prefers content → description → name). Applied
in this commit.

### F-D3 🟡 `mnema search` observation hit has no key, no title — just an id

**Observed:** Observation hits come back as `{ entity: "observation",
id: "...", key: null, title: null, snippet: "..." }`. The CLI
`formatHit` renders `entity ' '\n  snippet`, which looks visually
abandoned ("`observation `" — empty bold key).

**Root cause:** observations genuinely have no first-class identifier
beyond the UUID. `parentKey` is null (no FK to task in the indexed
shape). The renderer treats `null` keys as empty strings.

**Severity:** medium — cosmetic, but the hit is hard to act on.

**Decision:** punt to a doc note rather than fix structurally.
Observations are append-only ephemera; a UUID prefix in the snippet
header would be the right call once we touch the renderer next.
Tracked in `docs/TECH_DEBT.md`.

### F-D4 🟡 `mnema doctor --rebuild-mirrors` skips every other check

**Observed:** `mnema doctor --rebuild-mirrors` runs only the rebuild
path, then exits — it does not print the config/version/db/migrations
checklist. The `--help` does not mention this.

**Severity:** medium — UX asymmetry. The expectation from `--flag`
options is "do everything plus this", not "instead of this".

**Decision:** acceptable for v0.3, document in CHANGELOG. The
operation is targeted (recover missing mirrors) and the user typically
runs plain `mnema doctor` first to see the drift warning, then
`--rebuild-mirrors` to act on it. A separate `mnema mirror rebuild`
subcommand may be a better long-term home — defer that until enough
recover flows are needed.

### F-D5 🟢 Skill mirror frontmatter lacks `created_at` / `updated_at`

**Observed:** Memory `.md` mirror has `created_at` + `updated_at`,
skill `.md` mirror does not (only `last_used_at`). Asymmetric for
humans reading both.

**Severity:** low — cosmetic consistency.

**Fix:** added both fields to the skill mirror frontmatter. Applied
in this commit.

### F-D6 🟢 Process: same version, different tarball

**Observed:** A `saurim-mnema-0.3.0-alpha.1.tgz` from before migration
009 was sitting on disk. The Camada-2 commit added the file without
bumping the version, so a second `pnpm pack` produced a tarball with
the same name and different content (8 vs 9 migration files).

**Severity:** low for now (alpha, single-developer iteration) but
high for any future "share the tarball" workflow.

**Fix:** none in code. Convention reminder: when something user-visible
changes between alpha taggings, bump to `-alpha.N+1`. Tracked under
`docs/TECH_DEBT.md` §9 — schema/migrations conventions.

### Cross-scenario findings (not bugs)

- **Drift warning loud (F-A)**: confirmed verbatim. `pino` log line
  at `level:40 (warn)` with the pending filename, message references
  `mnema migrate`. Cleanly separable from `level:30` boot info line.
  No regression.
- **FTS5 latest-skill-only deduplication**: confirmed. Two skill rows
  with same slug, different versions — only v2 surfaces in search.
  The latest-version JOIN in `searchSkills` works as designed.
- **Diacritic insensitivity**: `migracao` ≈ `migração` across all
  three new tables, courtesy of `unicode61 remove_diacritics 2`. No
  surprises.
- **rebuild-mirrors idempotency**: confirmed. Three calls in a row
  yielded "nothing to rebuild" on the second and third.
- **30 observations**: search ran in ~400 ms (most of it node boot).
  No regression, no warnings.
- **decision↔memory cross**: search returns both hits, distinguished
  via `entity` field. No dedup, which is correct — they're separate
  artefacts with different semantics.

## What worked well (no friction)

- The audit log captures every CLI mutation with the right `kind` and
  `actor`. dev6 working `MNEMA_ACTOR=dev6 mnema ...` produced clean
  entries with `actor: 'dev6'` throughout.
- `mnema migrate` applied migration 009 with one line of output and
  zero ambiguity. The fact that dev6 hit this naturally (DB at 8, disk
  at 9 after the pack swap) is a real-world rehearsal of the recovery
  path from F-1.
- `--rebuild-mirrors` reported the affected slugs by name on each call:
  good signal-to-noise. No `--quiet` needed.

## What dev6 found non-obvious

- **`per-entity limit` truncates silently.** Default 25 hits per
  entity; with seven matching observations the user sees 25 of N
  without any "(N more truncated)" hint. Worth adding once volumes
  grow. Tracked.
- **Audit log does not capture raw SQL inserts.** Expected, but the
  drift warning loud (F-A) is the *only* defense if an agent ever
  bypasses the MCP/CLI to write to SQLite directly. The boot warning
  earned its keep here.
- **FK constraints on `created_by`** for skills/memories/observations
  bit during seed scripts. The service path runs
  `identity.ensureActor` first; raw SQL has to do it manually. Not a
  bug, but a documentation opportunity for "how to seed a Mnema DB"
  flows.

---

## Recommendation

Three fixes applied in the same commit as this eval (`--entity`
plumbing, memory/skill snippet column choice, skill mirror
timestamps). Two acceptable trade-offs documented (rebuild-mirrors
skipping main checks; observation hit shape).

This feels like the right cut line. 0.3.0-alpha.2 or 0.3.0-rc.1 with
the three CLI fixes would be a solid candidate for socialisation once
that's the right move. Not pushing for it.

Open items for the next phase:
- Search result rendering polish (per-entity truncation banner,
  observation hit visual treatment).
- "Seed scripts / SQL utility" doc note for advanced users who want
  to bulk-load test data.
- Process: version bump discipline when migrations or shipped JS
  changes between tarballs.
